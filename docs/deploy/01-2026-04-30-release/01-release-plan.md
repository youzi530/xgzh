## 3. 灰度策略 (Rollout)

### 3.1 为什么要灰度

首次发版 = 用户量 0 → N. 任何上线后第一次接到真实流量的 bug 都会放大成事故. 灰度的意义是 **每一档观察期内, 只有 5% / 10% / 50% 的真实用户暴露**, 出事可立即回退到旧版本影响面可控.

微信小程序的 "灰度" 不是金丝雀 (k8s 那种), 而是**版本分发比例**: 后台勾 "灰度 5%" 后, 微信会用 hash 把 5% 的活跃用户路由到新版本, 其它 95% 仍用线上正式版. 用户主动 "重启微信" 或 "卸载重装" 不影响这个比例.

### 3.2 灰度档位

| 档位 | 比例 | 观察窗口 | 通过条件 (全满足才放下一档) | 回退动作 |
|------|------|---------|-------------------------|----------|
| Stage 1 | **5%** | ≥ 30 min | 错误率 < 1% / 0 红线词触发 / 0 死链 / 0 5xx 突增 / DingTalk 无 P0 告警 | "停止灰度" → 等同回到 0% |
| Stage 2 | **10%** | ≥ 30 min | 同上 + 5% 阶段累计 0 用户投诉 (微信小程序后台 → 用户反馈) | 同上 |
| Stage 3 | **50%** | ≥ 60 min | 同上 + Sentry issue P1+ ≤ 1 | 同上 |
| Stage 4 | **100%** | — | 全量稳定 | "回退到旧版本" (要前一版仍存档) |

> ⚠️ **首次发版没有 "旧版本" 可回退** — 0% → 5% → 10% → 50% → 100% 这条单行道, 任何 P0 只能 **暂停灰度** + **修复重新提审 + 重新发** (3 工作日审核窗口). 这是首次发版最大的风险, 见 [`04-rollback-plan.md`](./04-rollback-plan.md) §"首次发版无旧版本"专章.

### 3.3 监控指标 (灰度期间必看)

| 指标 | 阈值 | 来源 | 超阈处理 |
|-----|------|------|---------|
| API 5xx 占比 | < 1% | `error_monitor` 滑窗 60s + DingTalk 告警 | P0: 立即回退 |
| AI 红线词触发率 | = 0 | `tests/test_forbidden_pattern_filter.py` 监控点 + Sentry breadcrumb | P0: 立即回退 (合规阻塞) |
| 用户登录成功率 | > 95% | `/api/v1/admin/dashboard?days=1&format=html` "登录成功率" | P1: 暂停灰度查 root cause |
| 微信支付拉起失败率 | < 5% (本次为 stub, 应 = 0%) | 同上 dashboard | P1: 同上 |
| Quota 误命中 (免费用户) | < 1% | dashboard "AI quota" 段 | P2: 观察 |
| 用户反馈 (微信平台) | 0 P0 投诉 | 微信小程序后台 → 用户反馈 | 看具体内容 |

监控面板入口:

- 后端 dashboard: `https://api.<你的域名>/api/v1/admin/dashboard?days=1&format=html`, 带 `X-Admin-Token` header (`OPS_ADMIN_TOKEN` 环境变量)
- Sentry: `https://sentry.io/organizations/<你的 org>/issues/?environment=prod`
- DingTalk 告警群: 通过 `ALERT_DINGTALK_WEBHOOK` 配置, 每 60s 错误率 ≥ 1% 自动 `@ oncall`

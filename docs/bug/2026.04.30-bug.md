**lessons learned (v2 retro 5 条新条目, 追加到 spec/25)**:

1. **ACR 个人版控制台教程页是 single source of truth** — 创建仓库后直接看 控制台 → 镜像仓库 → 教程, 直接拿 ``docker login`` / ``docker push`` 命令的精确格式, 不要凭文档假设. 个人版 / 企业版 / 不同区域 域名都不同. spec 写法也要从 ``教程`` 页 copy
2. **pydantic-settings 默认 env var 命名是 field_name → FIELD_NAME (大写, 无前缀)** — 想加前缀必须显式 ``model_config = SettingsConfigDict(env_prefix="XGZH_")``. 历史代码没加 → spec 也不能想当然加 ``XGZH_`` 前缀. 容器 env var 名必须跟 BE 实际读的一致
3. ``/healthz`` healthy ≠ 业务 OK — healthcheck 不查 DB 是设计良好 (DB 慢不该让 readiness 红), 但反过来意味着 healthcheck 通不能证明 DB 连得上. 调试连接问题必须**主动跑业务 endpoint** (随便一个 list articles 之类)
4. **`docker rm <pg-container>` 前必须 `docker inspect` 看 mount** — `docker volume ls` + `docker inspect --format '{{.Mounts}}'` 5 秒钟事, 但救一次能省一次数据事故. 如果 mount type 是 ``bind`` 或 ``volume`` 才安全 rm; 没 mount 就是数据在容器 layer 里, rm 即丢
5. **手 docker run 的容器跟 docker compose 不互通** — compose 用 label / project name 标记自己管的容器, 手起的容器没标记 → compose recreate 时识别为 "orphan" 会留着, 但同名 create 时会冲突报 ``Container name already in use``. 解决: 不要 docker run + docker compose 混用; 全部走 compose, 或第一次切换时手动清干净裸容器

## 三、CPA/CPS 券商返佣闭环（核心收入）

### 3.1 返佣模式说明

| 模式 | 定义 | 单价区间（参考） | 数据来源 |
|------|------|-------------|---------|
| **CPA**（Cost Per Acquisition） | 注册/开户即返佣 | ¥30-300/人 | 基本款 |
| **CPS**（Cost Per Sale） | 入金/交易后分成 | 入金额 0.5%-3% / 佣金分成 30%-70% | 富途、老虎、长桥多采用 |
| **混合** | 开户 ¥X + 入金返点 Y% | 富途常见 ¥800-2000/有效开户 | 高价值 |
| **阶梯** | 月新增 N 人后单价升级 | - | 业绩激励 |

### 3.3 转化追踪与归因

```
用户路径：
  XGZH App → 点击券商卡片 → 跳转券商 H5/App（带 utm_source=xgzh & invite_code=XXX）
       ↓
  ① 券商方记录"来自 xgzh"
       ↓
  用户完成 注册 → KYC → 入金 → 交易（事件链）
       ↓
  ② 我方通过【两条路径】对账：
       a. 券商 API 主动回调（CSV 报告 / Webhook）
       b. 用户回到 XGZH 上报开户成功（自助绑定，可发 ¥10 立减）
       ↓
  ③ 业务后台看板对账，T+30/60 与券商财务结算
       ↓
  ④ 给推荐用户发激励（影响 LTV）
```

### 3.4 转化率与 ARPU 预估（保守模型）

```
假设：
- DAU = 5,000  → MAU ≈ 25,000  → 注册用户 ≈ 60,000（第 12 个月）
- 券商详情页点击率 = 8%（DAU × 30%日活打开 × 8% = 120 次/天）
- 点击 → 注册转化率 = 12%（券商页面）= 14 次/天
- 注册 → 入金转化率 = 35% = 4-5 次/天有效入金
- 单笔有效开户 CPA + CPS = ¥800（保守）
  → 日收入 ≈ 14 × ¥800 × 35% = ¥3,920
  → 月收入 ≈ ¥117,600
  → 年化 ≈ ¥1,400,000
```

### 3.5 自建返佣后台（管理端必备）

```sql
-- 每日报表视图
CREATE VIEW v_broker_daily AS
SELECT 
    broker_id,
    DATE(created_at) AS day,
    COUNT(*) FILTER (WHERE event_type = 'click') AS clicks,
    COUNT(*) FILTER (WHERE event_type = 'signup') AS signups,
    COUNT(*) FILTER (WHERE event_type = 'kyc_pass') AS kyc_passes,
    COUNT(*) FILTER (WHERE event_type = 'deposit') AS deposits,
    SUM(amount_cny) FILTER (WHERE event_type = 'deposit') AS deposit_amount
FROM conversion_events
GROUP BY broker_id, day;

-- 用户层归因
CREATE TABLE attribution (
    user_id          UUID,

    first_click_at   TIMESTAMPTZ,
    last_click_at    TIMESTAMPTZ,

    deposit_at       TIMESTAMPTZ,
    deposit_amount   NUMERIC(20,2),
    cpa_received     NUMERIC(10,2),
    cps_received     NUMERIC(10,2),
```

### 6.6.1 UGC 社区审核 SOP（Sprint 6 增补 / DOC-S6-001）

> 背景：Sprint 6 引入 UGC 社区（发帖 / 评论 / 点赞 / 举报），用户输入比 LLM 输出更脏，必须在 BE-S5-001 红线词词典之上增加用户输入侧二级审核 + admin 审核队列。本节是法务签字底稿。

#### 用户协议条款（注册时再次勾选）

- 「我同意《社区规则》及《用户协议》第 N 条："不得发布任何形式的承诺收益、推荐买入、内幕信息、私域引流（微信号 / QQ / 群号 / 二维码）、个人证件号 / 手机号 / 银行卡号"」
- 不勾选不得发帖 / 评论。条款全文产品上线前由律师定稿。

#### 三级处罚机制

| 违规等级 | 触发条件 | 处罚 | 申诉窗口 |
|---------|--------|------|--------|
| **首次违规** | Tier 1 内容（必涨 / 加微信 / 内幕等）命中 1 次 | 内容自动 reject + 站内信 warn + 不影响账号 | 24h 站内申诉 |
| **二次违规** | 同账号 7 天内 Tier 1 命中 ≥ 2 次 | 7 天禁言 + 历史违规帖批量 self_only 化 | 24h 站内申诉，admin 复审 |
| **三次违规** | 同账号 14 天内 Tier 1 命中 ≥ 3 次，或一次发布严重违法内容 | 永久封号 + 注销账户 + 注册手机号黑名单 | 7 天邮件申诉 (legal@xgzh.app) |

#### 审核工作流（先审后发）

```
用户提交 → audit_user_content (v3) →
  ├─ Tier 1 / 私域引流 / 隐私数字串命中 → status=rejected (visibility=self_only) → 即时站内信
  ├─ Tier 2 命中 → status=pending → 进 admin 队列 (24h SLA) → 超时自动 published
  └─ 全过 → status=published
```

- **admin SLA 24h**：钉钉群每日 09:00 提醒昨日待审帖；超时自动通过（避免阻塞用户体验）
- **批量举报阈值**：同一帖被举报 ≥ 5 次自动 hidden 等 admin 复审，不立即 reject
- **admin 复审 4 选项**：approve（恢复 published）/ reject（永久 rejected）/ hidden_continue（保持 hidden 等用户申诉）/ delete（永久删，可选连带封号）

#### 数据留存与申诉

- **rejected / hidden / deleted 帖子**：保留 90 天用于申诉（用户可在"我的发布"看到），90 天后真删
- **举报记录**：保留 1 年用于复盘恶意举报模式
- **审核轨迹**（reviewed_by / reviewed_at / rejection_reason）：永久存档用于监管核查
- **用户申诉**：admin 后台 24h 内回复；驳回需附详细理由

#### 反 spam 限流策略（BE-S6-009）

- 发帖 60s ≤ 1 / 24h ≤ 10
- 评论 10s ≤ 1 / 24h ≤ 50
- 点赞 1s ≤ 5（防快速点赞刷榜）
- 举报 60s ≤ 1 / 24h ≤ 5（防恶意举报）
- 新用户 7d 内只读（user.created_at < now - 7d）：防黑产用大批量新号灌水
- 注销过的账户：永久禁止再发帖（防 burner 账户）

### 6.7 风险预案（极端情况处置）

| 场景 | 预案 |
|------|------|
| 监管约谈 | 准备《产品说明书》《合规承诺函》《免责声明全文》随时提供 |
| 第三方投诉版权 | 24h 内下架对应内容 + 邮件回复 + 复盘 |
| AI 输出造成用户损失 | 完整对话日志存证；以用户协议免责条款应对；保留法律顾问 |
| 券商合作中止 | 提前下架链接 + 通知用户 + 寻找替代券商 |
| 数据泄露 | 应急小组 + 72h 通报 + 重置密钥 + 用户通知 |
| **UGC 社区被恶意刷帖 / 引流** | admin 队列 + Redis 限流 + 黑产 IP 黑名单（超 5 次违规自动加），24h 应急方案：临时关闭社区发帖入口（仅保留浏览） |
| **UGC 用户上传违法内容** | admin 1h 内删除 + 封号 + 留存证据备查；如涉刑事内容 → 联系网信办举报中心 |
